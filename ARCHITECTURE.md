# Architecture Diagram: Static IaC Evaluation Pipeline

This document describes the technical mechanics of the AWS SAM validation orchestrator, the compiler constraints addressed, and the structure of the physical evaluation pipeline.

## Problem Statement

Large Language Model generation for Infrastructure-as-Code fails against deterministic standards due to three constraints:
1. API Surface Inconsistencies: Models omit mandatory properties required by strict `cfn-lint` verification rules.
2. Environment-Blind Generation: Instructions lack explicit bounds, resulting in hallucinated syntax unaligned with AWS Serverless Application Model schemas.
3. Unactionable Validation Loops: Static code review by an LLM creates prediction anomalies where invalid code is approved because physical compiler traces are not provided.

## Core Mechanics

### Physical Compilation Baseline
The pipeline avoids language-model-based reviews. Feedback is exclusively generated by sequential static execution runs against the `yaml.safe_load`, `cfn-lint`, and `cfn-guard` libraries. This forces the optimization layer to anchor exclusively against deterministic runtime exceptions.

### Parameterization via MIPROv2
The instruction sets generating the YAML structures are treated as internal hyperparameters. These parameters are tuned using Bayesian optimization (MIPROv2) evaluating output directly against the static penalty function.

## Component Design

### Prompt Generator (src/factory.py)
The orchestrator drives a DSPy ChainOfThought schema wrapped within a `PromptFactory` object. It ingests an architectural target and localized AWS SAM structure definitions to output an explicitly bounded instructional parameter sheet.

### Evaluator Pipeline (src/evaluators.py)
Student language models ingest the prompt and return raw YAML. The YAML is written to disk and processed across three sequential subprocess commands:
1. YAML Structure Verification: The core `yaml.safe_load` utility parses the data format natively (+0.20 weight).
2. Specification Enforcement: The `cfn-lint --format json` module validates adherence to CloudFormation parameter topologies (+0.40 weight).
3. Security Compliance Validation: The `cfn-guard validate --output-format json` mechanism asserts constraints like explicit S3 KMS encryption blocks are fully integrated natively (+0.40 weight).

### Bayesian Optimizer Loop
The system incorporates DSPy 3.1.3 MIPROv2 optimization. The pipeline calculates a localized validation float by evaluating metric execution outputs minus iteration cost parameters. Candidate instructions are algorithmically rejected when `cfn-lint` thresholds fail repeatedly.

### ChromaDB Vector Grounding (src/data_loader.py)
Local directories contain pre-configured AWS SAM definitions mapping known compilation constraints.
When the Evaluator Pipeline captures an explicit system execution traceback (e.g., `[Lint E3002] Resource properties missing`), the error code is mapped to the `sam_declarative_reference` database. A discrete snippet of AWS topology is extracted and embedded sequentially into the context window for error correction looping.

### Execution Isolations
Tests execute locally via mocked `unittest.mock.patch` injection bypassing native binary dependency structures in continuous integration loops.

## Known Limitations & Future Work

1. Runtime Testing Constraints: The system evaluates infrastructure statically by verifying that code successfully parses a CloudFormation graph via `cfn-lint`. It does not provision physical resources to AWS nor evaluate if those architectures function during live operational deployment. This architecture strictly validates declaration schema, not physical networking behavior.
2. Binary Dependencies: Currently, `cfn-lint` and `cfn-guard` must be explicitly installed by the runner within the `$PATH`. Production pipelines require automated artifact retrieval scripts embedding these dependencies inherently within `requirements.txt` processes.
